1. MapReduce概览

MapReduce是谷歌提出的在分布式系统上并行处理大数据的一种框架，由Jeffrey Dean在论文“MapReduce: Simplified Data Processing on Large Clusters”中提出。其大致想法如下：

6.824的lab 1是实现一个map reduce的word counter，我们的输入是若干文章，每一个文章以(k1,v1)表示，其中k1是filename，v1是content。master会将文章分为nMap份，并将其交给nMap个worker去做。这一任务分配并非同时进行，而依赖于go语言的channel机制。

经过map function之后，我们得到结果list(k2, v2)，这里k2代表word，v2代表word出现的次数。所有worker的输出被分成nReduce份，存在磁盘里。文件名可以记为“mr-X-Y.json”，其中1<=m<=nMap，1<=n<=nReduce。

等到所有map都做完之后，再进行reduce操作。由于同一个word会在不同的map result中出现，我们会将其归纳起来得到一个list(k2, list(v2))，通常在这一步之前需要对k2进行排序增加效率。接着调用reduce function得到(k2, v2)，也就是把同一个单词出现的次数加起来，存在结果中并综合起来。

具体而言，map和reduce function的结果如下

    reducef func(string, []string) string

这两个函数是已经写好的直接调用即可。

我们需要模仿这样的代码去完成我们的任务。

一个问题在于，我们并不是一开始就将nMap个任务分给worker，而是需要一个动态的分配操作。

具体而言，worker需要首先在master那里register获得一个workerID，之后需要向master请求任务，任务发送过来之后，需要根据任务的Phase(map or reduce)来调用对应的方法。如果在操作过程中出现错误，例如整个worker失效，则需要返回错误信息并将worker初始化；如果成功完成，也需要向master报告结果。

（实际上经过我测试发现代码错误的问题就在rpc通信AskTask这一块，目前定位了bug但还不知道为什么错了。目前事情比较多，如果助教做过这个lab希望能询问一下可能出错的原因。）

master的调度较为复杂，首先我们初始化master的时候需要定义一个容量为max(nMap,nReduce)的channel TaskChan便于任务分发（吐槽一句go居然没有针对int的max），其次还需要一个Task的列表TaskStates实时更新所有任务的状态。首先我们会将所有Task初始化为MapState，然后开始每隔一段时间检查全体任务是否已经完成，若否，则开启一个schedule的goroutine，一次检查TaskStates里Task的状态，大致上来说，对于超时/错误/初始化的任务会将任务放出TaskChan重新分配，并将其状态设置为TaskWaiting，即在TaskChan中等待分发。

具体分发过程依赖于和worker.go里函数的配合。即需要写出每一个worker call的master函数，具体而言包括worker在maaster中的CallRegister/Register，worker向master获得任务的CallAskTask/AskTask，以及worker向master汇报信息的CallReportInfo/ReportInfo。注意到这里函数需要适当加锁，即

xxxxxxxxxx
Master.mu         sync.Mutex
m.mu.Lock()
defer m.mu.Unlock()

但注意AskTask，即从TaskChan中分发Task的函数是不能加锁的。此外ReportInfo之后还需要开一个goroutine去检查schedule master中TaskStates。

func ihash(key string) int {
    h := fnv.New32a()
    h.Write([]byte(key))
    return int(h.Sum32() & 0x7fffffff)
}
